🔬 5/15 GTM 発火状況 実測結果（B⑤ 検証完了） BigQuery 実測ベース
⚠️ 真因確定：GTMコンテナ GTM-5B3L5372 は サイトに正しく読み込まれているものの、scroll_depth / cta_click / form_start / form_abandon の4イベントは 5月全期間 BigQuery に1件も到達していません。contact_finish（送信完了）のみ計18件正常発火しています。

■ 検証手法

  • BigQuery analytics_386840839.events_* を _TABLE_SUFFIX BETWEEN '20260422' AND CURRENT_DATE() でフルスキャン
  • 各イベント名の日別カウントを集計（5/1〜5/14 / 14日分）
  • 同期間内の event_params から scroll_pct / cta_* / form_* カスタムパラメータ受信状況も並列確認
  • 本番サイト www.ark-hd.co.jp/ および /contact/ の HTML を取得し、GTM スニペット読み込みとフォーム要素の data-tracking 属性の有無を実測

■ 真因仮説（GTM 公開済み × BQ 0件 の差分）

  1. HTML側マッチ要素の不足：本番サイト /contact/ のフォームは旧来型 <form name="form1"> 構造で、GTMタグのトリガー条件として一般的な data-tracking 属性 / data-form-name 属性が付与されていません。タグは公開されていても、トリガーが マッチする要素 が存在しない状態です。
  2. Scroll Depth 組み込みトリガーの未有効化の可能性：GA4 構成タグの「Scroll Depth」組み込みトリガー（25/50/75/90%）が GTM 管理画面で無効化されている、または該当タグに紐付けされていない可能性があります。
  3. CTAクリック の Click Trigger 条件のミスマッチ：CTAクリック収集の Click トリガーが data-tracking 属性ベースで設計されている場合、HTML側に同属性が存在しないとマッチしません。

■ contact_finish のみ正常発火している理由

送信完了ページ /contact/?mode=finish の URL パスをトリガー条件とした Page View 系タグ（contact_finish）は、HTML 側の要素属性に依存せず URL のみで発火条件を判定できるため、正常に動作しています。これにより「お問い合わせ件数」だけは正確に計測できています（5/6 sessions=22, submissions=2 → CVR 9.09% の数値整合は確認済み）。

単位ルール（5/13 整合性レビュー対応で追加）

Looker側で「同じKPIなのに値が違って見える」事象を防ぐため、BQ側の指標列に _pct 接尾辞を導入しました。

列名単位用途
engagement_rate素値（0.xx）計算用・GA4標準準拠
engagement_rate_pctパーセント（0〜100）Looker スコアカード・グラフ表示用
overall_cvr素値（0.xx）計算用
overall_cvr_pctパーセント（0〜100）Looker 表示用（問合せ + 資料DL）
inquiry_only_cvr_pctパーセント（0〜100）送信のみCVR（submissions ÷ sessions × 100）
クライアント直観値と完全一致
⚠️ 二重100化に注意: Looker側で _pct 列を使う場合、書式は「数値」のままにしてください。「%」書式を当てると 9.09% が 909% と表示される事故が発生します。
